Beginner
OOP là gì?
Lập trình hướng đối tượng (Object-Oriented Programming) là paradigm lập trình tổ chức code xung quanh đối tượng — thực thể kết hợp dữ liệu (thuộc tính) và hành vi (phương thức). Đây là cách mô hình hóa thế giới thực vào phần mềm.
Beginner
Class vs Object
Class là bản thiết kế (blueprint). Object là thực thể được tạo từ bản thiết kế đó.

public class Dog {
    private String name;
    public Dog(String name) {
      this.name = name;     // thuộc tính
    }
    public String bark() {
      return "Woof!";       // phương thức
    }

    public static void main(String[] args) {
      Dog rex = new Dog("Rex");  // object (instance)
    }
  }
Beginner
Constructor & Destructor
Constructor (__init__ / new) được gọi khi tạo object để khởi tạo trạng thái. Destructor (__del__ / finalizer) được gọi khi object bị hủy để giải phóng tài nguyên.
Beginner
Instance vs Class vs Static members
Instance: mỗi object có bản sao riêng (self.x). Class: dùng chung cho mọi instance (@classmethod). Static: không phụ thuộc class hay instance, chỉ là hàm tiện ích nằm trong class (@staticmethod).
4 trụ cột OOP
Encapsulation
1. Đóng gói (Encapsulation)
Gói dữ liệu và phương thức vào một đơn vị, ẩn chi tiết nội bộ. Dùng access modifier để kiểm soát quyền truy cập.

public class BankAccount {
    private int balance = 0;      // private

    public void deposit(int amount) {  // public interface
      if (amount > 0) {
        balance += amount;
      }
    }

    public int getBalance() {
      return balance;
    }
  }
Lợi ích: giảm coupling, dễ thay đổi nội bộ mà không ảnh hưởng code ngoài.
Abstraction
2. Trừu tượng (Abstraction)
Chỉ expose những gì cần thiết, ẩn đi sự phức tạp. Thực hiện qua abstract class và interface.

public abstract class Shape {
    public abstract double area();     // interface, không có implementation
  }

  public class Circle extends Shape {
    private double r;
    public Circle(double r) { this.r = r; }
    @Override
    public double area() { return 3.14 * Math.pow(r, 2); }
  }
Inheritance
3. Kế thừa (Inheritance)
Class con (subclass) kế thừa thuộc tính và phương thức từ class cha (superclass). Thúc đẩy tái sử dụng code.

public class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public String speak() { return "..."; }
  }

  public class Cat extends Animal {
    public Cat(String name) { super(name); }
    @Override
    public String speak() { return "Meow"; }
  }

  public class Dog extends Animal {
    public Dog(String name) { super(name); }
    @Override
    public String speak() { return "Woof"; }
  }
Lưu ý: ưu tiên composition over inheritance khi mối quan hệ không rõ ràng là is-a.
Polymorphism
4. Đa hình (Polymorphism)
Cùng một interface, nhiều hành vi khác nhau. Có 2 dạng chính:

Compile-time (overloading) — cùng tên method, khác tham số (Java/C++).
Runtime (overriding) — subclass override method của parent.

import java.util.Arrays;
  import java.util.List;

  List animals = Arrays.asList(new Cat("Kitty"), new Dog("Rex"));
  for (Animal a : animals) {
    System.out.println(a.speak());  // Meow / Woof — runtime polymorphism
  }
Dùng với abstract class/interface để viết code linh hoạt, không cần biết kiểu cụ thể.
SOLID Principles
S
Single Responsibility Principle
Một class chỉ nên có một lý do để thay đổi. Mỗi class chịu trách nhiệm cho một chức năng duy nhất.
❌ Class User vừa xử lý auth, vừa render HTML, vừa gửi email.
✅ Tách thành UserAuth, UserEmailService, UserView.
O
Open/Closed Principle
Class nên mở để mở rộng, đóng để sửa đổi. Thêm tính năng mới bằng cách thêm code mới, không sửa code cũ.
Thực hiện qua abstract class, interface, Strategy pattern.
L
Liskov Substitution Principle
Object của subclass phải có thể thay thế object của superclass mà không làm hỏng chương trình.
❌ Square extends Rectangle vi phạm LSP vì thay đổi width cũng đổi height.
Preconditions không được mạnh hơn, postconditions không được yếu hơn class cha.
I
Interface Segregation Principle
Client không nên bị buộc phụ thuộc vào interface mà nó không dùng. Tách interface lớn thành nhỏ hơn.
❌ Interface Animal có method fly() — Dog sẽ phải implement method vô nghĩa.
✅ Tách thành Flyable, Swimmable, Runnable.
D
Dependency Inversion Principle
Module cấp cao không phụ thuộc module cấp thấp — cả hai phụ thuộc vào abstraction. Abstraction không phụ thuộc detail; detail phụ thuộc abstraction.
Thực hiện qua Dependency Injection (DI).
Design Patterns phổ biến
Creational
Singleton
Đảm bảo chỉ có một instance của class trong toàn bộ chương trình. Dùng cho: DB connection, config manager, logger.
getInstance() trả về cùng một object mỗi lần.
Creational
Factory Method & Abstract Factory
Factory Method: định nghĩa interface tạo object, để subclass quyết định tạo class nào.
Abstract Factory: tạo họ các object liên quan mà không chỉ định class cụ thể. Dùng khi cần swap toàn bộ theme/platform.
Structural
Decorator
Thêm hành vi cho object tại runtime mà không thay đổi class gốc, bằng cách wrap nó. Dùng thay cho inheritance khi muốn kết hợp linh hoạt. 
Coffee coffee = new SimpleCoffee();
  coffee = new MilkDecorator(coffee);
  coffee = new SugarDecorator(coffee);
  System.out.println(coffee.cost());  // 3.5
Behavioral
Observer
Định nghĩa quan hệ one-to-many: khi subject thay đổi trạng thái, tất cả observer được notify tự động. Nền tảng của event systems, reactive programming.
Behavioral
Strategy
Đóng gói một họ thuật toán, cho phép swap thuật toán tại runtime. Loại bỏ conditional logic lớn. Ví dụ: các strategy thanh toán (Credit, PayPal, Crypto).
Structural
Proxy, Adapter, Facade, Composite
Proxy: đại diện cho object khác, kiểm soát truy cập (lazy loading, caching, security).
Adapter: chuyển đổi interface không tương thích.
Facade: cung cấp interface đơn giản cho hệ thống phức tạp.
Composite: tổ chức object thành cấu trúc cây, xử lý uniform.
Chủ đề nâng cao
Advanced
Composition vs Inheritance
Inheritance = quan hệ is-a (Cat is-a Animal). Tạo coupling chặt giữa class cha/con.
Composition = quan hệ has-a (Car has-a Engine). Linh hoạt hơn, dễ test hơn, ít side effect.
Rule of thumb: "Favor composition over inheritance" — GoF Gang of Four.
Advanced
Coupling & Cohesion
Coupling: mức độ phụ thuộc giữa các module. Mục tiêu: loose coupling — dễ thay đổi độc lập.
Cohesion: mức độ các thành phần trong module liên quan đến nhau. Mục tiêu: high cohesion — mỗi module làm tốt một việc.
Advanced
Mixin & Multiple Inheritance
Mixin là class nhỏ cung cấp tính năng cụ thể để mix vào các class khác mà không thiết lập quan hệ cha/con. Python dùng MRO (Method Resolution Order — C3 linearization) để giải quyết diamond problem trong multiple inheritance.
Advanced
Duck Typing & Structural Subtyping
"If it walks like a duck and quacks like a duck, it's a duck." Python không kiểm tra kiểu, chỉ kiểm tra method/attribute tồn tại. TypeScript dùng structural typing thay vì nominal typing.
Insane
Covariance, Contravariance & Invariance
Quy tắc về tính tương thích của generic types:
Covariant: List[Cat] là subtype của List[Animal] (read-only).
Contravariant: ngược lại (write-only).
Invariant: không có quan hệ subtype. Java arrays covariant nhưng generics invariant.
Insane
Metaprogramming & Metaclass
Metaclass là "class của class" — kiểm soát cách class được tạo ra. Dùng để: auto-register subclasses, enforce constraints, implement ORM (Django Models), create DSL. Trong Python: type là metaclass mặc định; custom bằng class Meta(type).
Insane
Prototype-based OOP
JavaScript dùng prototype chain thay vì class-based. Object kế thừa trực tiếp từ object khác qua [[Prototype]]. class syntax trong ES6 chỉ là syntactic sugar trên prototypes. Hiểu prototype chain là nền tảng để debug JS sâu.

Beginner
Sự khác nhau giữa class và object là gì? ▶
Class là bản thiết kế (blueprint) định nghĩa cấu trúc và hành vi. Object là instance cụ thể được tạo từ class đó, tồn tại trong bộ nhớ và có trạng thái riêng. Một class có thể tạo ra nhiều object.
4 tính chất của OOP là gì? Giải thích từng cái. ▶
Encapsulation: ẩn dữ liệu nội bộ, expose qua interface. Abstraction: ẩn sự phức tạp, chỉ show những gì cần. Inheritance: class con kế thừa từ class cha. Polymorphism: cùng interface, nhiều hành vi khác nhau.
Interface vs Abstract class khác nhau thế nào? ▶
Abstract class: có thể có implementation, thuộc tính, constructor. Class chỉ kế thừa một class (đơn kế thừa). Dùng khi có code chung.
Interface: chỉ định nghĩa contract (method signatures). Class có thể implement nhiều interface. Dùng khi muốn định nghĩa khả năng (capability). Java 8+ cho phép default methods trong interface.
Overloading vs Overriding? ▶
Overloading (compile-time polymorphism): cùng tên method trong một class, khác số lượng/kiểu tham số. Quyết định lúc compile.
Overriding (runtime polymorphism): subclass định nghĩa lại method của parent với cùng signature. Quyết định lúc runtime qua virtual dispatch.
Intermediate
SOLID là gì? Cho ví dụ vi phạm một nguyên tắc. ▶
Xem tab SOLID. Ví dụ vi phạm SRP: class Report vừa tính toán dữ liệu, vừa format output, vừa gửi email → có 3 lý do để thay đổi. Khi thay đổi cách gửi email, phải sửa class Report dù logic report không đổi.
Composition over inheritance — tại sao? ▶
Inheritance tạo coupling chặt: thay đổi class cha ảnh hưởng tất cả con. Dễ dẫn đến "fragile base class problem". Composition linh hoạt hơn — có thể swap behavior lúc runtime, dễ test hơn (mock dependencies). Ví dụ: thay vì FlyingFish extends Fish extends Animal, dùng Fish has SwimmingBehavior và FlyingBehavior.
Singleton pattern — ưu nhược điểm, khi nào dùng? ▶
Ưu: đảm bảo một instance, tiết kiệm tài nguyên, global access point.
Nhược: global state khó test (unit test khó isolate), vi phạm SRP, thread-safety phức tạp, tạo hidden dependencies.
Dùng khi: DB connection pool, logger, config manager, cache. Cân nhắc Dependency Injection thay thế để code testable hơn.
Dependency Injection là gì? Tại sao quan trọng? ▶
DI là pattern inject dependencies từ bên ngoài thay vì tạo trong class. 3 dạng: constructor injection, setter injection, interface injection.
Lợi ích: loose coupling, dễ test (inject mock), dễ swap implementation, tuân thủ DIP trong SOLID. Frameworks như Spring, Angular, .NET Core built-in DI container.
Advanced
Liskov Substitution Principle — ví dụ vi phạm thực tế? ▶
Classic example: Square extends Rectangle. Rectangle có setWidth() và setHeight() độc lập. Nhưng Square phải giữ w=h, nên setWidth(5) cũng đổi height thành 5 — vi phạm kỳ vọng của client dùng Rectangle. Solution: tách Square và Rectangle, không có quan hệ kế thừa.
Giải thích diamond problem và cách ngôn ngữ giải quyết. ▶
Khi class D kế thừa B và C, cả B và C đều kế thừa A — method của A bị duplicate. Python dùng C3 linearization (MRO) để xác định thứ tự tìm method. Java/C# tránh bằng cách không cho multiple inheritance class, chỉ cho multiple interface. C++ cho phép nhưng cần virtual inheritance.
Sự khác biệt giữa shallow copy và deep copy? ▶
Shallow copy: tạo object mới nhưng các thuộc tính reference vẫn trỏ cùng object. Thay đổi nested object ở copy → ảnh hưởng original.
Deep copy: clone toàn bộ object tree. Hoàn toàn độc lập. Python: copy.copy() vs copy.deepcopy(). Prototype pattern dùng deep copy.
Cohesion và Coupling — tại sao cần high cohesion, low coupling? ▶
High cohesion: module tập trung làm một việc tốt → dễ hiểu, maintain, reuse. Low coupling: ít phụ thuộc giữa module → thay đổi một module không cascade ra module khác, dễ test độc lập, dễ scale team. Đây là metric đo "good design" trong OOP.
Insane
Covariance vs Contravariance trong generic types? ▶
Covariant (out T / ? extends T): producer — chỉ đọc. List[Cat] có thể gán cho List[Animal]. Safe vì chỉ read.
Contravariant (in T / ? super T): consumer — chỉ ghi. List[Animal] có thể gán cho List[Cat].
Invariant: Java generics mặc định. Kotlin dùng in/out keywords rõ ràng. Remember: PECS — Producer Extends, Consumer Super.
Metaclass trong Python hoạt động như thế nào? ▶
Mọi class trong Python là instance của type. Khi Python thấy class Foo:, nó gọi type('Foo', bases, dict). Custom metaclass override __new__ hoặc __init__ của type để intercept class creation. Use cases: Django ORM dùng metaclass để register model fields, enforce singleton, auto-implement methods. Ví dụ: class Meta(type): def __new__(mcs, name, bases, ns): ...
Virtual dispatch (vtable) hoạt động thế nào ở cấp độ runtime? ▶
Mỗi class có một vtable (virtual method table) — mảng con trỏ đến các virtual methods. Mỗi object có con trỏ đến vtable của class nó. Khi gọi virtual method, runtime tra cứu vtable → gọi đúng implementation của subclass. Đây là cơ chế runtime polymorphism ở C++/Java. Chi phí: thêm một level indirection (dereferencing pointer), rất nhỏ nhưng tồn tại.
Thiết kế một hệ thống thực tế — áp dụng OOP principles? ▶
Ví dụ: Payment system. (1) Abstraction: PaymentProcessor interface với charge(), refund(). (2) Implementations: StripeProcessor, PayPalProcessor. (3) DIP: service nhận PaymentProcessor qua constructor injection. (4) OCP: thêm CryptoProcessor không sửa service. (5) SRP: tách PaymentLogger, FraudDetector riêng. (6) LSP: mọi processor thay thế được nhau. Dùng Strategy pattern để swap processor.
